Critical Application Examples

Specific instances of AI use and critical evaluation

Learning SVG Through Guided Instruction

Context: I wanted to add a lock icon to the Privacy page but didn't know how to create SVG graphics programmatically.

My Approach:

  • Asked Claude to explain SVG coordinate systems, viewBox, and basic shapes first
  • Requested detailed breakdown of path commands (M, L, Q, C, Z) with visual examples
  • Only after understanding the fundamentals did I ask for lock icon implementation
  • Asked Claude to annotate the code explaining why each path command was chosen

Critical Evaluation:

  • When Claude's first lock design looked too complex, I requested simplification—I needed clean lines that would render clearly at small sizes
  • Modified opacity values Claude suggested (0.3) to better balance with my existing design system
  • Asked Claude to explain corner decoration SVGs, then designed my own variation for card corners

Outcome: I now understand SVG well enough to create and modify decorative elements independently. The learning process was more valuable than the code output.

Solving Grid Layout Issues

Context: Cards on the About page weren't aligning properly in the grid, and accordions needed to span different column widths.

My Diagnosis: I identified that some cards needed to span 2 columns while others span 1, but CSS grid wasn't behaving as expected. I provided screenshots showing the misalignment.

Claude's Role: After I explained the desired layout (e.g., "Who We Are should span 2 columns, Non-Commercial should span 1, CCI should span 2"), Claude provided the correct `grid-column: span 2` syntax and explained how CSS Grid's auto-placement algorithm works.

Critical Moment: When Claude suggested using flexbox instead, I rejected this because it would conflict with the existing grid system used across all pages. Consistency mattered more than solving this one layout problem with a different approach.

Spacing Helper Classes

Problem: I needed consistent spacing across pages but didn't want to write inline styles everywhere.

Request: I asked Claude to generate utility classes for my 8-point spacing system (--space-1 through --space-24).

Critical Evaluation: Claude generated margin and padding helpers (.m-t-4, .p-y-6, etc.). I reviewed each class to ensure it followed my existing naming conventions and didn't conflict with component-specific spacing. These utility classes significantly accelerated layout adjustments.

Implementation Page: Design Through Dialogue

Context: The Implementation Rationale page needed to explain design decisions to instructors.

Iterative Process:

  1. Framework Discussion: I outlined what content sections I wanted (color system, typography, components, process). Claude proposed a structure.
  2. First Draft Rejection: Claude wrote comprehensive paragraphs for every section. I called this "too boring" and "too much text without visual interest."
  3. Requirements Refinement: I specified: "Show actual color swatches, render typography examples in their actual fonts, display button sizes visually. Use accordions to hide detailed explanations."
  4. Content Revision: I rejected the "Design Process" section that focused on my background. Requested rewrite focusing on design thinking principles instead.
  5. Visual Format: I asked for the final Design Principles section to use "quote cards" format—something not used elsewhere on the site—to create visual interest.

Outcome: The final page structure emerged through negotiation. Claude proposed; I critiqued; Claude revised. This back-and-forth created a page that balanced information density with visual presentation—a design decision AI couldn't make independently.

Voice AI for User Simulation

Context: I couldn't conduct live user testing with older adults, but needed to stress-test whether my design decisions addressed their needs.

Method: Used ChatGPT voice to role-play conversations:

  • "Pretend you're Akasha, 67 years old with limited tech literacy. I'll describe the interface, and you tell me what confuses you."
  • "Act as Denisho, who needs to access this site privately at work. Would you feel safe clicking the 'Join Community' button?"

Value: These simulations exposed assumptions I hadn't questioned. For example, "ChatGPT-as-Akasha" asked why there were four ways to access Community—did that mean they were different features? This validated my redundancy strategy but also revealed I needed clearer labeling.

Mock Code Interviews

Purpose: Practice explaining my code and design decisions under pressure.

Setup: I asked ChatGPT voice to interview me: "You're a senior developer reviewing my code. Ask me why I made specific implementation choices."

Example Questions ChatGPT Asked:

  • "Why did you choose CSS Grid over Flexbox for the card layout?"
  • "Your buttons use min-height: 48px. Where does that number come from?"
  • "I see you're using font-weight: 500 for body text instead of 400. Justify that decision."

Value: Forced me to articulate the 'why' behind every decision, revealing gaps in my understanding that I then researched properly.

AI Limitations Encountered

Where AI fell short and human judgment prevailed

No Visual Understanding

AI cannot "see" layout problems. When cards misaligned or spacing looked off, I had to diagnose the issue first—identify the specific CSS property causing problems—before AI could help fix it. Screenshots helped communicate problems, but AI couldn't independently identify visual design flaws.

Diagnosis Required

Design Judgment Absence

AI proposed technically correct solutions that violated design principles. When I wanted to add visual interest to the Implementation page, Claude suggested adding images. But what images? Of what? AI couldn't determine that color swatches, font specimens, and button size comparisons would be more meaningful than decorative photos.

Lacks Context

Over-Engineering Tendency

Claude frequently suggested complex solutions when simple ones sufficed. I had to repeatedly specify "use existing component classes" rather than accepting AI's tendency to write new, custom CSS for every situation. Maintaining system consistency required constant vigilance.

Simplicity Enforced

Reflection on AI-Assisted Development

What this experience revealed about human-AI collaboration

AI tools accelerated implementation but didn't replace design thinking. Every significant decision—color palette, typography system, component hierarchy, spacing strategy—originated from my analysis of Phase 3 research and design training. AI helped me execute those decisions efficiently, explained technical concepts I didn't understand, and generated code scaffolding I could then refine.

The most valuable skill wasn't prompt engineering—it was critical evaluation. Knowing when to reject AI suggestions required understanding accessibility standards, CSS architecture, semantic HTML, and user needs well enough to recognize when outputs were technically correct but contextually inappropriate. AI cannot determine whether a solution serves project goals, respects user dignity, or aligns with ethical principles. That judgment remains fundamentally human.

The learning-first approach proved essential. By asking AI to explain concepts before generating code, I built understanding that made me a better evaluator of AI outputs. I didn't just get working code—I learned how to write, debug, and improve it independently. AI became a teacher and assistant, not a replacement for learning.
