Learning-Centered Process

How I learned from AI rather than copying from AI

Step 1: Understanding Before Implementation

Principle: Never request code without first understanding the underlying concept.

Example - Learning CSS Grid:

  • Wrong approach: Give me CSS Grid code for a 3-column card layout.
  • My approach: Explain how CSS Grid's auto-placement algorithm works. What's the difference between explicit and implicit grid tracks? When should I use grid-template-areas vs grid-column spans?
  • Then I asked: Now show me an example, but annotate every line explaining why that property is necessary.
  • Result: When cards misaligned later, I could diagnose the problem myself (missing grid-column span on specific items) rather than asking AI to fix it blindly.

Example - SVG Path Commands:

  • Asked Claude: Explain M, L, Q, C, Z commands with visual diagrams. What do the numbers mean in M20,30 vs L40,50?
  • Only after understanding coordinate systems did I request a lock icon for the Privacy page
  • When the result was too complex, I knew why it was complex (too many curve control points) and could specify: Use straight lines and one quadratic curve maximum
  • Later designed book, sun, and brain SVGs independently by applying the same principles
Step 2: Evaluating AI Output Against Standards

Principle: Every AI suggestion must be verified against professional standards and project requirements.

Example - Accessibility Fixes:

  • Cursor flagged 36 WCAG violations. I asked it to explain each one's impact on real users, not just the technical fix
  • For color contrast issues, verified Cursor's suggestions using WebAIM's contrast checker independently
  • Rejected Cursor's first focus outline suggestion (blue) because it clashed with my green accent system—requested alternatives aligned with design tokens
  • Cross-referenced every ARIA attribute recommendation with W3C documentation to ensure semantic correctness

Example - Design System Consistency:

  • Claude suggested unique hover effects for each page. I rejected this—consistency across pages was more important than novelty
  • When ChatGPT proposed adding animations, I evaluated against motion-sensitivity guidelines and rejected most for being unnecessary
Step 3: Saying "No" to AI

Principle: Critical thinking means knowing when to reject AI suggestions.

Times I Said No:

  • Claude suggested Flexbox for About page layout—rejected because Grid was the established pattern
  • Cursor proposed multiple font families for visual interest—rejected to maintain typographic hierarchy
  • ChatGPT recommended adding profile photos to testimonials—rejected as potentially stigmatizing for mental health context
  • Claude's first three SVG designs for Resources page were too abstract—requested concrete book icon instead
Step 4: Iterative Refinement Through Dialogue

Principle: The first AI output is never the final solution. Design emerges through critique and iteration.

Example - Badge Visual System:

  1. Iteration 1: Claude suggested flat badges. I critiqued: Too visually boring, needs subtle depth.
  2. Iteration 2: Added gradients and shadows. I rejected: Shadow too strong, looks outdated. Make it subtle.
  3. Iteration 3: Refined gradient stops and inset shadows. I requested: Add non-color differentiation for accessibility—green badges bold, yellow badges have different border.
  4. Iteration 4: Tested in high-contrast mode, discovered borders weren't visible. Added explicit high-contrast media query styles.
  5. Final: A badge system that works visually and accessibly, developed through 4+ rounds of critique.

Key Insight: AI provided rapid iteration capacity, but every design decision came from me. The speed of iteration was AI's contribution, not the design judgment itself.

Step 5: Using Voice AI to Test Design Rationale

Principle: If you can't explain your design decisions verbally, you don't understand them well enough.

Method: ChatGPT Voice role-play sessions:

  • User simulation: You're Akasha, 67, struggling with tech. I describe the interface, you tell me what's confusing. Revealed my navigation redundancy wasn't as clear as I thought—needed better signposting.
  • Client presentation: You're the client, skeptical of my design choices. Challenge every decision. Forced me to justify color choices beyond I like green—led me to research color psychology in healthcare contexts.
  • Code interview: Ask me technical questions about my implementation. When I couldn't explain why buttons were 48px tall, I researched touch target accessibility—learned about WCAG 2.5.5 Target Size.

Value: Voice interaction prevented copy-paste entirely. Gaps in understanding became immediately obvious when I couldn't articulate reasoning verbally.

Step 6: Applying Learning Independently

Principle: The ultimate test of learning is applying concepts without AI assistance.

What I Built Independently:

  • Modified Cursor's SVG designs for Resources and Activity pages without asking for help
  • Debugged spacing issues on Personal Space page by applying CSS Grid knowledge independently
  • Created custom color variations for high-contrast mode after understanding the pattern Cursor taught me
  • Wrote this entire acknowledgement page structure without AI input—used AI only for grammar checking

Critical Reflection: AI Limitations & What I Learned

Understanding AI's fundamental constraints and my responsibilities as a designer

No User Empathy or Design Vision

AI cannot understand what it feels like to be Akasha, 67, fearing stigma while seeking mental health support. AI also cannot evaluate visual hierarchy, balance, or aesthetic coherence. Every significant decision—color psychology, navigation redundancy, privacy language—originated from empathy mapping and user research.

Learning: Design thinking remains irreplaceable. AI implemented my vision but couldn't conceive it.

Ethical Blindness

AI doesn't understand ethical implications. When Claude suggested profile photos for testimonials, I had to recognize this could stigmatize mental health seekers.

Learning: Ethical responsibility cannot be delegated.

Over-Engineering

AI tends toward complexity. Claude generated custom CSS when existing components would work. Use existing design tokens became my most frequent instruction.

Learning: Simplicity requires human discipline.

False Confidence Risk

AI-generated code looks professional, creating a dangerous illusion of correctness. I almost shipped Claude's accessibility implementation without verification—until I cross-checked with W3C docs and found incorrect ARIA usage.

Learning: AI's authoritative tone can suppress critical thinking. I treat every AI output as a first draft requiring verification against authoritative sources.

Declaration of Authorship

All design decisions, ethical considerations, and critical evaluations in this project originated from my own analysis and judgment. AI tools served as implementation assistants and learning facilitators, but they did not author the design thinking, user research synthesis, or accessibility strategy that shaped this platform.

Where AI suggested code, I evaluated its appropriateness, modified it for context, and verified its correctness against authoritative sources. This website reflects my design education, ethical reasoning, and technical learning—amplified by AI tools but not replaced by them.
