Implementation Rationale

From research insights to visual design: How Phase 3 findings shaped Phase 4 implementation decisions.

Core Design Philosophy

Restraint as strategy—accessibility as foundation, clarity as craft

My design process prioritized readability and accessibility as foundational requirements, with aesthetic refinement following as a secondary layer. This wasn't a compromise—it was a recognition that information design, like poster design, relies on contrast, hierarchy, and visual clarity. The research from Phase 3 revealed that older adults like Akasha need interfaces that reduce cognitive load, not add to it. Every design decision emerged from this principle: make information accessible without sacrificing visual interest.

Professional Standards & Compliance

Adhering to ACS Code of Ethics and WCAG 2.2 AA accessibility standards

WCAG 2.2 AA Compliance

This platform adheres to WCAG 2.2 Level AA accessibility standards, ensuring:

  • Color contrast ratios of 4.5:1 for normal text and 3:1 for large text
  • Touch targets meeting 48×48px minimum (WCAG 2.5.5)
  • Proper semantic HTML with ARIA labels for screen readers
  • Keyboard navigation support for all interactive elements
  • High contrast mode compatibility for visual impairments
  • Non-color visual differentiation (bold, borders, icons)

Chromatic Restraint

Color acceptance varies dramatically across users, particularly among older adults who may have color vision changes or strong preferences. I chose a warm black-and-white foundation (smoky black #0b0a07 and isabelline off-white #fdf9f4) because neutrals provide universal accessibility while establishing a calm, approachable atmosphere. The cream background reduces eye strain compared to pure white, particularly important for extended reading sessions.

Accent colors—dark spring green (#04724d) and maize yellow (#ffe76d)—serve as visual punctuation, not paragraphs. Green and yellow are relatively universal colors with broad cultural acceptance. The zaffre purple (#3626a7) primary color adds creative distinction while the green and yellow accents provide chromatic balance, preventing the purple from dominating. This triadic relationship creates visual interest without overwhelming users who might find saturated color palettes distracting or stressful.

Functional Color Usage

Colors serve functional roles: green indicates positive actions and trustworthy information (badges like "Active Community", hover states), yellow highlights important but non-critical information ("Private & Secure", "Updated Weekly"), and purple anchors primary navigation and focus states. This semantic consistency helps users build mental models—green means "safe to proceed", yellow means "pay attention", purple means "current location or interactive element".

Typography: Warmth Meets Structure

Three typefaces balancing readability with personality

Card Titles & Accordion Headers

Google Sans Code (Monospace) — h3, accordion headers

Playful, slightly technical personality—reminiscent of American Typewriter but with contemporary digital clarity. The monospace rhythm creates predictable character width, aiding scanning and comprehension. Adds gentle structure and a hint of whimsy.

Section Titles & Buttons

Comfortaa (Rounded Sans) — h1, h2, blockquotes, buttons

Rounded letterforms convey warmth, approachability, and safety. In a mental health context, sharp serifs or geometric sans-serifs can feel clinical; Comfortaa's soft curves create psychological comfort. Bold enough for emphasis but never aggressive.

The workhorse typeface for all body text, set at font-weight 500 (medium) rather than the standard 400 (regular). On the cream background, the slightly heavier weight improves contrast and readability without appearing bold. This decision prioritizes legibility for users with declining vision—Akasha shouldn't need to squint. The 18px base size and 1.7 line-height create generous vertical rhythm, reducing eye strain during extended reading.

Component Design Decisions

Thoughtful choices behind cards, badges, buttons, and accordions

Button Sizing & Touch Targets

48×48px
60px height

All interactive elements meet or exceed 48×48px touch targets, with primary buttons sized at 48×48px or 60×60px. This isn't just WCAG compliance—it's pragmatic recognition that users with arthritis, tremors, or simply larger fingers need generous hit areas. The larger sizing also ensures button text remains fully legible at a glance.

WCAG 2.5.5 Accessible Generous
Card System as Clickable Links

Entire cards are wrapped in anchor tags, making the full card surface clickable rather than limiting interaction to a single button. This decision emerged from recognizing unpredictable user behavior—some users will click the headline, others the description, others will aim for a button that may not exist. By making the entire card interactive, I maximize the probability of successful navigation while maintaining clear visual hierarchy through hover states and focus indicators.

This approach also reduces decision fatigue. Users don't need to locate a specific "Learn More" button; anywhere they click on the card advances them toward their goal. For users like Akasha with limited tech literacy, this forgiving interaction model prevents the frustration of "clicking the wrong place" and having nothing happen.

Badge System

Badges distill information into scannable labels—"Active Community", "Evidence-Based", "Private & Secure". This serves users who don't want to read full paragraphs but need quick confirmation of key attributes. The badge system does information subtraction, removing cognitive load while preserving essential meaning. Color-coding (green for positive/safe, yellow for attention/important, purple for general categorization) creates visual patterns that aid recognition across pages.

Accordion Pattern

Accordions progressively disclose information, giving users control over their experience. Rather than forcing users to consume all content immediately (information overload), accordions let them choose what to explore. This respects cognitive capacity and attention span—particularly important for older adults who may feel overwhelmed by dense pages. Accordions also preserve visual purity; the closed state maintains clean, breathable layouts while the open state provides depth for engaged users.

Information Architecture & Strategic Redundancy

Multiple pathways ensure users find what they need

Navigation Structure

Primary functions live in the top navigation (Community, Resources, Personal Space), while meta-informational pages reside in the footer (About, Privacy, Implementation). This separation creates a clear mental model: the top navigation answers "What can I do?", while the footer answers "Who are you and how does this work?". This purely functional approach emerged from my undergraduate design research experience in New York and graduate IT systems training—merging design thinking with information systems architecture.

Modular CSS Architecture

The CSS is deliberately split into semantic modules: nav.css, footer.css, components.css, base.css, layout.css. This modular approach enables component reusability and maintainability. Each file serves a distinct purpose—base.css establishes the design system (colors, typography, spacing), components.css defines reusable UI elements (buttons, cards, badges), and page-specific files handle unique layouts.

Unpredictable User Behavior

Users don't follow designer-prescribed paths. On the homepage, users can access the Community feature through: (1) top navigation menu "Community", (2) hero section "Join the Community" button, (3) "Community Discussions" card in the approach section, (4) "Get Started" button in the CTA section. Each entry point serves users with different browsing patterns. This isn't lazy design—it's intentional redundancy that ensures critical functions remain accessible regardless of how users navigate.

Visual Balance & Rhythm

SVG decorations and spacing systems create hierarchy

SVG Decorations

Pure text and utilitarian interfaces risk becoming monotonous, potentially causing users to disengage. However, excessive decoration creates visual noise. SVG decorations—wave dividers, lock icons, corner embellishments—strike a balance. They add personality and visual rhythm without demanding attention or conveying functional meaning that users might misinterpret.

On the About page, subtle wave dividers create sectional breaks. On the Privacy page, the lock icon immediately communicates security through universal symbolism. These elements use the primary color with reduced opacity (0.3-0.6), ensuring they remain decorative rather than dominant.

Spacing System

The spacing system uses an 8-point scale (--space-1: 4px through --space-24: 96px), creating visual consistency and rhythm. However, I don't follow it dogmatically. Information design is about grouping—certain elements need to be visually clustered to signal relationships. When standardized values would create awkward gaps, I adjust. The system provides guardrails, not prison walls.

Privacy-First Technical Decisions

This platform deliberately avoids common web practices like analytics tracking, geolocation APIs, and third-party integrations. Users like Denisho face workplace discrimination; Fligo risks family rejection; Akasha is vulnerable to scams. For them, privacy isn't convenience—it's safety. We don't collect location data, even though it could personalize resources. We don't embed analytics scripts. We don't use localStorage or persistent cookies. Each of these technologies represents a potential vulnerability. Trust is built through restraint—by not collecting data we don't absolutely need, we signal respect for user autonomy and safety.

Design Principles

Core insights that guided every decision

"Design isn't decoration—it's information architecture."

Color, typography, spacing, and components form relationships that clarify or obscure meaning. Badges transform complexity into visual tokens. Consistency turns elements from decoration into signal.

"The hardest skill is knowing what to remove."

Restraint isn't minimalism—it's strategic editing. SVG decorations add breathing room but stay opacity-reduced. Visual interest serves user tasks, not ego.

"Systems serve content, not vice versa."

The spacing scale creates rhythm 90% of the time. The other 10% requires judgment. Good systems have clear rules and the wisdom to break them when necessary.

Accessibility Audit Evidence

IBM Equal Access Checker results and remediation

Audit Results Summary

Tool Used: IBM Equal Access Checker (browser extension) and WAVE Web Accessibility Evaluation Tool

Pages Audited: All 12 pages tested across multiple breakpoints (mobile, tablet, desktop)

Critical Issues Found: 0

Violations Requiring Fix: Initially 36 across all pages (primarily ARIA labels, contrast ratios, landmark structure)

Final Status: All Level 1 and Level 2 issues resolved. Level 3 recommendations implemented where feasible.

Key Fixes Applied:

  • Added sr-only labels for all search inputs (previously placeholder-only)
  • Wrapped form inputs in <fieldset> with <legend> for logical grouping
  • Corrected accordion headers from <div> to role="heading" with aria-level
  • Increased color contrast for discussion metadata from gray (#888) to dark gray (#333) — now meets 4.5:1 ratio
  • Added explicit @media (prefers-contrast: high) styles for badges, cards, and buttons
  • Wrapped quotations in <q> tags (12 instances across genai_mt_acknowledgement.html)
  • Added title attributes to complement ARIA labels for search inputs
WCAG 2.2 AA Zero Critical

Reflection

The IBM Equal Access Checker revealed patterns I hadn't anticipated: placeholder text, while visually clear, provides no semantic value to screen readers. This reinforced the importance of redundant labeling strategies.

Most impactful fix: wrapping form inputs in <fieldset> elements. Screen readers now announce "Reply to discussion" before reading each input field, providing critical context that was previously missing.

The audit also highlighted subtle issues like directional language ("left unaddressed" → "unaddressed") and the need for explicit high-contrast mode styles—both invisible in standard testing but crucial for users with specific needs.

Testing Process: Ran IBM Checker on every page after each significant update. This iterative approach prevented issue accumulation and made fixes easier to trace.

Challenges & Problem-Solving Strategies

Technical issues encountered and solutions implemented

Responsive Grid Layout

Problem: Cards misaligned when content varied—creating visual chaos on Resources page.

Solution: CSS Grid with grid-template-columns: repeat(auto-fit, minmax(300px, 1fr)) plus explicit grid-column: span for semantic hierarchy. Combined with align-items: start.

Learning: Auto-placement works for uniform content; explicit control needed for intentional spanning.

CSS Grid Solved

ARIA State Management

Problem: Screen readers announced inverted mobile menu state ("collapsed" when open).

Solution: Read aria-expanded first, then toggle: 

const isExpanded = 
  this.getAttribute("aria-expanded") === "true";
this.setAttribute("aria-expanded", !isExpanded);

Learning: Timing matters—read state before modifying DOM.

JavaScript ARIA

Progressive Enhancement

Problem: Accordion content flashed before JS loaded, causing layout shift.

Solution: Content visible by default → Add .js-enabled class to <html> → CSS hides content only when JS available → Accordion reveals on click.

Learning: Never assume JavaScript will load. Content must be accessible first.

Progressive Enhancement Solved

Focus Visibility

Problem: IBM Checker flagged invisible focus indicators—outline: none in reset.css removed them.

Solution: Use :focus-visible with explicit styles in base.css: 

button:focus-visible {
    outline: 3px solid var(--accent-green);
    outline-offset: 2px;
}

Impact: Zero focus-related issues across all pages.

Phase 2: Accessibility & Ethics Analysis

Work Phase 2 established the accessibility and ethical framework that guided all implementation decisions—from WCAG 2.2 AA compliance to ACS professional ethics.

The accessibility audit conducted in Phase 2 revealed critical insights about older adult users (particularly Akasha, 67) who require simplified navigation, larger touch targets, and clear semantic structure. Ethical analysis through the ACS lens emphasized public interest primacy—mental health support cannot prioritize aesthetics over accessibility. This principle manifested in features like profile privacy controls (respect & autonomy), transparent data usage documentation (honesty & trustworthiness), and multi-pathway navigation (reducing tech anxiety for vulnerable users). Every implementation choice was evaluated against the question: "Does this serve user wellbeing first?"
